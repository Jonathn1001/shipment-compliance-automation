The Mission & the Journey

What you built, why it matters to a logistics company, and the exact path a shipment takes through it — framed for a 30-minute interview on problem-solving and technical methodology.

The one-sentence pitch

Before a cargo shipment can cross a border, a human has to check that its paperwork is complete and self-consistent. This project automates the boring, error-prone part of that check.

It ingests shipment data, validates it against compliance rules, flags every problem, and produces a readiness report a person can act on.

🧭
Why Safiri cares

Global trade drowns in repeated manual checks, inconsistent data, and time-sensitive reviews. A module like this turns a 20-minute manual paperwork review into a 2-second automated one — with a human only for the cases that genuinely need judgment.

🎯
How to use this for a 30-minute interview

This walkthrough defends the two things the interview scores directly: problem-solving and technical methodology. Modules 2–6 give you the system knowledge for “walk me through X.” Module 7 — How I Worked targets the two axes head-on: decomposition, documented assumptions, testing method, systematic debugging, and the AI-usage note the rubric requires.

02

The Data Model

Five characters run the whole show. Knowing who does what is how you answer “walk me through your schema.”

A metaphor: the patient chart

A hospital keeps one master chart per patient — the single authoritative record. Lab reports arrive and update it, doctors write diagnoses, a discharge summary is produced, and nurses log every action with a timestamp that is never erased.

This module has exactly those five roles.

📦

Shipment

The master chart. The single source of truth, filled in over time.

📑

DocumentIngestion

A lab report that came in — stored raw and as clean data.

🚩

ValidationIssue

A diagnosis — one specific problem a rule found.

📄

ReadinessReport

The discharge summary — the decision, snapshotted.

🧾

AuditLog

The nurses’ notes — one timestamped line per action, never erased.

The Shipment is the source of truth

A shipment is created with almost nothing and gets completed over time. It carries a lifecycle — a small set of named states it can be in.

schema.prisma 
enum ShipmentStatus {
  CREATED
  READY
  NEEDS_REVIEW
  BLOCKED
  APPROVED
}
PLAIN ENGLISH

The complete list of states a shipment may hold…

just born — barely any data yet.

clean — safe to proceed.

a human must look before it moves.

a hard stop — it cannot be approved.

a human signed off — done.

Defining these as an enum means only these five values can ever exist.

🗂️
Say this in the interview

“Most shipment fields are nullable on purpose — a shipment starts as a near-empty shell and is progressively completed from documents. The rules then flag which required fields are still missing.”

Why documents get their own layer

A tempting shortcut is to let each document write straight onto the shipment row. That throws away two things the business actually needs.

🔍

Provenance

What did this document actually contain, and when? Each ingestion stores the raw payload untouched.

⚖️

Disagreement

When a document contradicts the record, that clash is itself a compliance signal — so it must survive, not be silently overwritten.

So each DocumentIngestion keeps both layers: rawInput (exactly as received) and mappedFields (the clean, canonical projection). This is the load-bearing decision of the whole design (documented in ADR-0001).

Some things are written once, forever

Two of the five characters are append-only. You never edit them; you only add new entries.

🕰️
Every validation run appends a new ReadinessReport

You get a timeline of how a shipment’s compliance posture changed — not just the latest verdict.

🔒
Every state-changing action appends one AuditLog entry

Created, ingested, field-updated, validated, approved — a tamper-evident trail auditors can trust.

💡
Key insight: append-only is a compliance superpower

In regulated domains, “how did we get here?” matters as much as “where are we?” Append-only history answers the first question for free. Overwriting would destroy the evidence.

03

Messy Docs → Clean Data

Real shipment documents are inconsistent. Two small pure functions tame them — and they are the answer to “how do you handle bad input?”

A metaphor: the universal travel adapter

Every country has a different plug shape, but your laptop needs one socket. A travel adapter accepts any plug and fits it to the one socket you have.

Shipment documents are the plugs. The same fact — the invoice’s value — shows up under a dozen different key names.

invoice_value→ maps to invoiceValue
total_value→ maps to invoiceValue
amount→ maps to invoiceValue
Invoice Value→ maps to invoiceValue

The Field Mapper is the adapter: any of these plugs, one canonical socket.

Step 1 — ignore the noise in a key

Before matching, every key is stripped down: lowercased, punctuation removed. That is why invoice_value, Invoice Value, and invoiceValue are treated as the same key.

field-mapper.ts 
const normalizeKey = (key: string): string =>
  key.toLowerCase().replace(/[^a-z0-9]/g, '');
PLAIN ENGLISH

Take any key name and boil it down…

make it all lowercase, then delete everything that is not a letter or a number. "Invoice Value" becomes invoicevalue — now it matches its cousins.

🔌
Why this is “semi-structured”-friendly

The take-home asked for structured and semi-structured input. Loose key-matching is exactly what makes the same mapper swallow tidy JSON, a slightly-off CSV header, and a hand-typed field alike.

Step 2 — turn text into the right type

A value like "1200" from a document is just text. A weight needs to be a number, a certification flag needs to be a true/false. One tiny function decides how to convert each field — driven by a single lookup table.

field-mapper.ts 
function coerce(field, value) {
  if (value === null || value === undefined) return undefined;
  switch (FIELD_KIND[field]) {
    case 'decimal': return toDecimalString(value);
    case 'int':     return toInt(value);
    case 'bool':    return toBool(value);
    default:       return toTrimmedString(value);
  }
}
PLAIN ENGLISH

Given a field and a raw value…

if there is nothing there, skip it.

Look up what kind of field this is…

money → keep exact precision as text,

a count → whole number,

a yes/no flag → true or false,

anything else → a trimmed string.

💡
One table to rule them all

FIELD_KIND is a single lookup of every field’s type. Adding a new field means adding one line there — the mapper, the database projection, and the update logic all read from it. That is “don’t repeat yourself” done right, and a great thing to point at when asked about maintainability.

Step 3 — reconcile without overwriting

Now the clean document data meets the shipment. The Reconciler makes one decision per field, and it never overwrites.

reconciler.ts 
for (const field of Object.keys(incoming)) {
  const documentValue = incoming[field];
  if (isEmpty(documentValue)) continue;

  const canonicalValue = current[field];
  if (isEmpty(canonicalValue)) {
    fills.push({ field, oldValue: undefined, newValue: documentValue });
  } else if (!valuesEqual(field, canonicalValue, documentValue)) {
    conflicts.push({ field, canonicalValue, documentValue });
  }
}
PLAIN ENGLISH

For each field the document supplied…

grab the document’s value.

If the document had nothing there, skip.

Now look at what the shipment currently holds.

If the shipment’s field was empty…

FILL it from the document.

Otherwise, if they disagree…

record a CONFLICT — but leave the shipment untouched.

The whole decision in three branches

Every incoming field lands in exactly one of these buckets.

1

Document empty?
Skip — nothing to do.

→
2

Shipment empty?
FILL from the document (and audit it).

→
3

They differ?
CONFLICT — keep the record, flag later.

🧪
Why “pure functions” keeps coming up

Both the mapper and reconciler are pure functions: no database, no network. That means the trickiest logic in the system can be unit-tested in milliseconds with plain objects — no test database required.

04

The Validation Engine

The heart of the system — and the part an interviewer will dig into hardest. Three ideas: pluggable rules, stateless recompute, and severity → status.

A metaphor: a panel of independent inspectors

Imagine a line of inspectors, each an expert in exactly one thing — one checks weights, one checks country codes, one checks wood-packaging certificates. Each raises a flag or stays silent. None of them knows or cares what the others do.

Hiring a new inspector does not require rebuilding the line. That is the single most important property of this engine.

🧩
The term for it: open/closed

The engine is open for extension, closed for modification. Adding a rule is adding a class — the engine code never changes. Say exactly that in the interview.

What one inspector looks like

Every rule is a class with one method: evaluate. It gets the shipment’s context and returns a list of problems — empty if all is well.

currency-iso.rule.ts 
evaluate(ctx: ValidationContext): IssueDraft[] {
  const { currency } = ctx.shipment;
  if (isBlank(currency)) return [];
  if (isKnownCurrency(currency)) return [];

  return [{
    issueType: 'currency-iso',
    severity: Severity.MEDIUM,
    field: 'currency',
    explanation: `Currency ...  is not a known ISO 4217 code.`,
    suggestedAction: 'Correct the currency ...',
  }];
}
PLAIN ENGLISH

Look at just the currency field.

If it is blank, this rule has no opinion — return nothing.

If it is a real currency code, also fine — return nothing.

Otherwise, raise one issue…

tagged with its rule-code,

a severity of MEDIUM (a warning, not a hard stop),

which field it is about,

a human-readable explanation,

and a concrete suggested fix.

🎚️
Notice the discipline

A missing currency is a different rule’s job. Each rule fires on exactly one concern and stays silent otherwise — that is what keeps 16 rules from stepping on each other.

The engine barely does anything

Because rules are self-contained, the engine that runs them all is almost trivial: hand each rule the context, collect every issue.

validation.service.ts 
const drafts: IssueDraft[] = this.rules.flatMap((rule) =>
  rule.evaluate(ctx),
);
PLAIN ENGLISH

Ask every registered rule to evaluate the shipment…

run each one’s evaluate,

and merge all their issues into one flat list.

The rules are injected as a list. Want a new check? Add a class to that list. The three lines above never change.

Stateless recompute — but waivers survive

Every run recomputes all rules from the shipment’s current state, then reconciles the fresh results against the old ones by a stable key: (issueType, field). Four things can happen to an issue.

✨

Created

A brand-new problem → inserted as OPEN.

🔁

Refreshed

Still a problem → kept OPEN, details updated.

✅

Resolved

Fixed since last run → marked RESOLVED, kept for history.

🛡️

Waived-kept

A human accepted the risk → never re-raised.

💡
The tension this resolves

Re-running must never show stale issues — yet a waiver a person granted must never be nagged about again. Recompute-plus-reconcile gives you both: fresh results, durable human decisions.

Issues in, a single verdict out

Once the issues settle, one pure function turns their severities into the shipment’s readiness status. No hidden state — just the issues.

status-resolver.ts 
export function resolveStatus(issues) {
  const hasCritical = issues.some((i) => i.severity === CRITICAL);
  if (hasCritical)
    return { status: 'BLOCKED', humanReviewRequired: true };

  const hasReviewable = issues.some(
    (i) => i.severity === HIGH || i.severity === MEDIUM);
  if (hasReviewable)
    return { status: 'NEEDS_REVIEW', humanReviewRequired: true };

  return { status: 'READY', humanReviewRequired: false };
}
PLAIN ENGLISH

Given the list of active issues…

is any of them CRITICAL?

If so → BLOCKED. A human must act; it cannot proceed.

Otherwise, is any HIGH or MEDIUM?

…

If so → NEEDS_REVIEW. A human must sign off.

Otherwise → READY. Clear to proceed.

🚧
Guarded approval

The same model protects the exit: a BLOCKED shipment cannot be approved — the API refuses with a 409 Conflict. The only way forward is to fix or waive the blocking issue and re-validate.

05

Production-Minded

“Beyond code that merely works” — the error handling, security, and tests that separate a strong submission from a demo.

A metaphor: a well-run building

A working building has walls and a roof. A safe building also has fire doors, keycard locks, an occupancy limit, and regular drills. None of that shows up in the brochure — but it is what keeps the building standing when something goes wrong.

🚪

Fire doors

The global error filter — one failure never leaks smoke (internals) everywhere.

🔑

Keycard locks

An optional auth gate on every route.

👥

Occupancy limit

Rate limiting and a body-size cap keep abuse out.

🧯

Fire drills

Tests that prove the system behaves under stress.

Every error speaks one language

Whatever goes wrong — a known domain error, a bad route, an unexpected crash — the client always gets the same shape: { error: { code, message, ... } }. Clients branch on a stable error code, never on fragile wording.

all-exceptions.filter.ts 
if (statusCode >= 500) {
  const detail = exception instanceof Error
    ? (exception.stack ?? exception.message)
    : String(exception);
  this.logger.error(summary, detail);
} else {
  this.logger.warn(summary);
}
PLAIN ENGLISH

If this is a real server error (500+)…

grab the underlying details — the stack trace if we have one…

and log it server-side so we can debug.

For ordinary client mistakes (4xx)…

just a one-line warning is enough.

🔐
Security by omission

The stack trace is logged, never sent to the client. Request bodies and raw document payloads are never logged. An attacker learns nothing from an error; an engineer learns everything from the logs.

Defense in depth

No single lock is trusted to do everything. Several independent layers each block a different class of abuse.

🪖
Helmet

Sets protective HTTP security headers on every response.

📏
64 KB body limit

An oversized payload is rejected before it is ever parsed or stored.

⏱️
Rate limiting (throttler)

Caps requests per client over a sliding window — blunts brute-force and denial-of-service.

✋
Validation pipe

Rejects unknown or malformed fields with a precise 422 and per-field details.

A tiny detail that signals seniority

When an auth token is configured, the guard compares it in constant time — so an attacker cannot guess the token character-by-character by timing responses.

api-auth.guard.ts 
function safeEqual(a: string, b: string): boolean {
  const ab = Buffer.from(a);
  const bb = Buffer.from(b);
  if (ab.length !== bb.length) return false;
  return timingSafeEqual(ab, bb);
}
PLAIN ENGLISH

Compare the provided token to the real one…

turn both into raw byte buffers,

bail early only on a length mismatch,

then compare with a function that takes the same time no matter what — no timing leak.

🎯
Know the limit of your own work

This guard proves who is calling (authentication). It deliberately does not decide which shipments a caller may see (authorization). Naming that boundary out loud is exactly the “documented assumptions” the rubric wants.

Tests that actually mean something

The test strategy is shaped like a pyramid: lots of fast pure-logic tests, fewer full end-to-end tests that exercise the real thing.

⚡
147 unit tests

Pure functions — mappers, reconciler, rules, status resolver — tested with plain objects in milliseconds, no database.

🗄️
29 end-to-end tests

The whole HTTP stack against a real, throwaway embedded Postgres — real requests, real transactions.

🖥️
17 frontend tests

The browser-side document parser and filename detection, so bad files fail fast before they hit the server.

💡
The interview soundbite

“The riskiest logic — validation and reconciliation — is pure, so it gets the fastest and most thorough tests. End-to-end tests then prove the wiring and the real database transactions actually hold together.”

The Architecture, End to End

Zoom out. The stack it runs on, the layers every request rides through, and the whole data flow in one picture — the answer to “walk me through your architecture.”

A metaphor: the subway network

A metro looks chaotic from the platform, but it is deeply ordered. Every train rides a fixed line through the same stations in the same sequence. A control center coordinates them — no train freelances its own route. The rails, signals, and rolling stock are standard parts, so you can swap a train model without re-laying track.

This system is built the same way. Every request is a train. It rides one line through an ordered set of stations, a central wiring layer coordinates who talks to whom, and the parts are standard enough to replace one without disturbing the rest.

There are three zones the trains run between.

🖥️
The Browser

A React single-page app. It parses files, draws the dashboard, and calls the backend over HTTP. It holds no business rules.

🛡️
The API

A NestJS server. Every rule, transaction, and decision lives here — behind a guarded, validated front door.

🗄️
The Database

PostgreSQL. The permanent record — shipments, documents, issues, reports, and an audit trail that is never erased.

🧭
Say this in the interview

“It is a three-tier app: a React SPA, a NestJS API that owns all the logic, and Postgres for durable state. The browser is deliberately dumb — nothing important happens there.”

The tech stack: what the network runs on

Nothing here is exotic. Every choice is boring, proven, and defensible — which is itself a senior signal. The backend uses pnpm; the frontend uses npm. Both are written in one language: TypeScript.

BACKEND

🏗️

NestJS 11

The framework. Gives structure, dependency injection, and clean seams for guards, pipes, and filters. Runs on Express under the hood.

🐘

PostgreSQL

The database. Chosen for real transactions, native enums, and exact Decimal money math — things a compliance system cannot fake.

🔌

Prisma 7 + pg adapter

The ORM. Talks to Postgres through a swappable driver adapter, so every query is parameterized and safe.

✔️

class-validator

Declares the exact shape each request must have. Unknown or malformed fields are rejected at the door with a 422, before any logic runs.

🛡️

helmet · throttler

Security headers and per-IP rate limiting, wired in globally. Cheap, standard defenses that a demo usually skips.

🧪

Jest + supertest

Unit tests for every pure function, plus end-to-end HTTP tests against a real throwaway Postgres. The safety net that made upgrades fearless.

FRONTEND

⚛️

React 19

Builds the UI from small components. The charts are hand-drawn SVG — zero charting libraries, zero bloat.

⚡

Vite 7

The bundler and dev server. In development it proxies /api straight to the backend.

🧭

React Router 7

Maps URLs to pages — dashboard, shipment list, shipment detail — without full page reloads.

🔬

Vitest

Unit tests the pure browser-side parsers — the code that turns a dropped CSV or JSON file into a clean payload.

💡
Key insight: boring is a feature

Every dependency here is mainstream and well-documented. When AI suggests an obscure library to solve a problem a standard tool already handles, that is your cue to push back. Fewer, proven parts means fewer surprises in production.

Every request rides the same rails

A single request does not jump straight to the logic. It passes through a fixed line of stations — each one a small, focused layer with a single job. This ordering is not decoration; it is what keeps the messy concerns (auth, validation, formatting, errors) out of your business code.

1
Guards

The rate limiter and auth guard decide if the request may proceed at all.

2
Validation Pipe

A pipe rejects unknown or malformed fields — the logic only ever sees clean input.

3
Controller

Thin. It reads the request, notes who is acting, and delegates. No business rules live here.

4
Service

The brain. Orchestrates the pure logic and wraps writes in a transaction.

5
Repository → Prisma → Postgres

The only code allowed to touch the database. Everything above it is database-agnostic.

6
Interceptor & Filter

On the way out, one interceptor shapes every success; one filter catches every error.

The powerful part: those cross-cutting layers are wired once, globally, and every route inherits them. You never repeat auth or validation code per endpoint.

app.module.ts 
providers: [
  { provide: APP_PIPE, useValue: new ValidationPipe(…) },
  { provide: APP_GUARD, useClass: ThrottlerGuard },
  { provide: APP_GUARD, useClass: ApiAuthGuard },
  { provide: APP_INTERCEPTOR, useClass: ResponseEnvelope },
  { provide: APP_FILTER, useClass: AllExceptionsFilter },
]
PLAIN ENGLISH

Here is the wiring for the whole app…

reject malformed input everywhere,

throttle abusive callers by IP,

require a valid token (checked in constant time),

wrap every success in a tidy { data } envelope,

turn every error into one predictable shape.

Declared once — every route gets all five for free.

🎯
Key insight: separation of concerns

Thin controllers, smart services, and repositories that alone touch the database. Each layer has one reason to change. This is why you can tell AI “put the rule in the service, not the controller” and know exactly what you mean.

Seams, not spaghetti

A few structural decisions are what let this system grow and change without becoming a knot. Each one creates a clean seam — a place you can cut and replace one part while everything around it stays put.

shipment.repository.ts 
update(id, data, tx: PrismaTx = this.prisma) {
  return tx.shipment.update({
    where: { id }, data,
  });
}
PLAIN ENGLISH

Update one shipment by its id…

run it against whichever client you are handed…

a plain connection by default,

or a transaction when the caller passes one.

One method, works alone or inside a bigger atomic write.

⚛️

All-or-nothing writes

Ingest and validate each wrap their writes in one transaction. A shipment never exists without its audit row; a half-finished run cannot corrupt state.

📖

Bounded pagination

Lists never load everything. Keyset pagination caps each page and stays fast at any depth.

🚦

Config validated at boot

Missing or malformed environment variables crash the app on startup with a clear message, not silently at 2am.

🔌

The database behind one service

Only PrismaService knows how to connect. That is why the Prisma 7 driver-adapter upgrade was a one-line change with every test still green.

🧩
Key insight: design for the swap

Good architecture is not about predicting the future — it is about isolating each risky part behind a seam so that when it does change, the blast radius is one file. Transactions, the repository pattern, and the single database service are all seams.

How I Worked

The two things a 30-minute interview actually scores — how you break a problem down, and how you work — plus the AI-usage note the rubric requires.

This interview is not a code review

Thirty minutes is too short to read code together. So the room probes two things instead: problem-solving — how you turn a vague brief into a plan and handle the parts it does not spell out — and technical methodology — how you actually build, test, and use your tools. The code already exists; this is about the thinking behind it.

Decomposition: skeleton first, then muscle

A metaphor: you do not build a house room-by-room, fully finished. You raise the frame of the whole house first — foundation, walls, roof — then fit out each room. A thin end-to-end skeleton proves the shape before any single part is polished.

The brief lists a six-step workflow. The trap is to build it as horizontal layers (all the models, then all the services, then all the endpoints) and have nothing that runs until the very end. Instead it was sliced into vertical slices — each one shippable on its own.

1
Walking skeleton

Create a shipment + append one audit row, end to end, with a test. The whole spine in miniature.

2
Ingest & map

Take mock document JSON, normalize keys, fill empty fields. Now data flows in.

3
Validate

The rule engine and severity ladder — the highest-risk, highest-value logic gets its own slice.

4
Report & harden

Readiness report, then the production concerns — errors, security, tests — layered on a working core.

💡
Say this in the interview

“I built vertical slices, not horizontal layers — a create-plus-audit skeleton first, then ingest, then validation. I always had something that ran end-to-end, so every later step was a small, testable addition instead of a big-bang integration at the end.”

The spec is vague on purpose — so decide, and write it down

A brief that says “suspicious invoice value” without a number is testing whether you notice the gap, make a defensible call, and document it. Guessing silently is the junior move; naming the assumption is the senior one — and it is a scored rubric line.

💰

“Suspicious” invoice value

Gap: no threshold given. Call: flag non-positive or implausibly out-of-range values as a warning, not a hard block — a human confirms. Why: false positives here are cheap; false blocks stall real cargo.

📅

“Allowed review window”

Gap: how old is too old? Call: made it a configurable REVIEW_WINDOW_DAYS, not a magic number. Why: the policy is a business setting, so it belongs in config, not baked into a rule.

⚖️

Gross vs. net weight

Gap: exact-equal, or a tolerance? Call: gross below net is a hard error; a small WEIGHT_TOLERANCE_PCT absorbs rounding. Why: real documents round; the rule must not cry wolf.

🔢

HS code validity

Gap: format only, or real classification? Call: validate the format now; treat a real WCO lookup as a documented, mocked seam. Why: the brief says mocked snapshots are fine — and honesty about scope beats faking a data source.

📝
The pattern that scores

Every one of these is: name the gap → make a call → give the reason → name what would change it. That four-beat structure is the same one that defends a trade-off — it is the shape of engineering judgment.

Methodology: test the risk first, commit in small steps

The method was not “write everything, then test.” The riskiest logic — the pure mappers, the reconciler, the rules, the status resolver — was written test-first. Each rule began as a red test describing the exact issue it should raise.

THE LOOP 
// 1. RED — describe the behavior, watch it fail
expect(runRule(shipmentWithGrossBelowNet))
  .toContainIssue('GROSS_LT_NET', 'high');

// 2. GREEN — simplest code that passes
// 3. REFACTOR — clean up, tests still green
PLAIN ENGLISH

First, write a test that says what a broken shipment should do…

assert the exact issue code and severity…

watch it fail for the right reason,

then write just enough code to make it pass,

then tidy up — the test proves you did not break it.

Because the risky logic is pure (no database, no framework), those tests run in milliseconds — which is why there are 147 of them. End-to-end tests then prove the wiring and the real transactions hold. Work landed in small, reviewable commits — a create-plus-audit slice, then ingest, then each rule — not one giant drop.

🎯
The interview soundbite

“I pushed the compliance logic into pure functions and tested it first. That is where the correctness risk lives, so that is where the fastest, densest tests are — and it kept the framework and database out of my most important decisions.”

Problem-solving in action: how I corner a hard bug

“Tell me about a tricky bug” is a problem-solving question in disguise. They are not testing the bug — they are testing your method. A senior method is boringly systematic, and it looks the same every time.

🔁
Reproduce first

No fix before a reliable repro. A pure-function design helps — feed it the exact input and the failure is deterministic.

✂️
Isolate

Halve the surface. Is it the mapper, the rule, or the transaction? Narrow until one component owns the failure.

💡
Hypothesize, change one thing

State what you think is wrong, change exactly one variable, and check — never shotgun several edits at once.

🔒
Lock it with a test

The fix is not done until a regression test would have caught it. The bug never comes back silently.

✎
Bring one real war story

Have a 60-second story ready that walks this exact arc on a bug you hit — [the reconciliation conflict that surfaced a wrong field, the rule that double-counted an issue, the transaction boundary you got wrong first]. Structure it: symptom → how I reproduced it → how I isolated it → the one-line cause → the regression test that now guards it. The arc is the point; the bug is just the vehicle.

A metaphor: the concept car

A concept car drives. It proves the design. But before it rolls off a production line it needs crash-testing, a supply chain, and manufacturing at scale. Nobody faults a concept car for not being a Camry — they judge whether the ideas are sound.

Your take-home is the concept car. The interview is where you show you know exactly what the production line would add.

🧭
The rubric’s real ask

“A credible path from take-home to production service” plus “documented assumptions.” Translation: name your trade-offs before they do, and have the next step ready.

The trade-offs, out loud

Four deliberate simplifications. For each: what it is, why it was fine here, and what would change it.

👁️

No real OCR

Why fine: the brief said mock data; a browser parser handles JSON/CSV/text. Next: the Field Mapper is already the clean seam to plug a real OCR service into.

🔁

Recompute all rules

Why fine: simple, correct, easy to reason about at this scale. Next: if the rule set or volume explodes, move to incremental evaluation — behind the same interface.

📊

Coarse severity ladder

Why fine: four levels give a clear, testable verdict. Next: swap the pure resolver for a score-based policy — rules and engine untouched.

🔓

Auth, not authorization

Why fine: a token gate closes the “world-callable” hole. Next: a user/tenant model for per-owner access — the biggest gap for multi-customer use.

Proof you can evolve it: the Prisma 7 upgrade

A newer database library removed the old way of connecting and required a driver adapter. The migration was surgical — one line in the connection setup — and every test still passed.

prisma.service.ts 
constructor(config: AppConfigService) {
  super({ adapter: new PrismaPg({
    connectionString: config.databaseUrl,
  }) });
}
PLAIN ENGLISH

When the database client is created…

connect it through the new Postgres driver adapter…

using the connection string from typed config.

One seam changed; the 176 tests were the safety net.

💡
Why this story lands

It shows you can absorb a breaking dependency change without fear — because the architecture isolated the database behind one service and the tests proved nothing else broke. That is production maturity.

The production roadmap

If Safiri said “ship it,” here is the order of work — highest-leverage first.

1
Per-owner authorization

A user/tenant model so a caller only sees their own shipments — the top gap for real multi-customer use.

2
Async ingestion + real OCR

Move heavy document processing onto a queue with background workers, so a slow scan never blocks a request.

3
Observability

Metrics, tracing, and dashboards so you can see rule failure rates and latency in production, not guess.

4
Richer policy engine

Replace the four-level ladder with configurable, per-customer compliance policies — without touching the rules.
